# Use Cases

## UC-1. Просмотр доступных VIP-столиков

### Связанные функциональные требования

* FR-1. Просмотр VIP-столиков
* FR-2. Просмотр доступности столика

### Связанные нефункциональные требования

* NFR-1. Время отклика системы не должно превышать 3 секунд
* NFR-4. Система должна быть доступна пользователям не менее 99% времени

### Актор

Клиент

### Цель

Получить список доступных VIP-столиков и свободных временных интервалов.

### Предусловия

* Система доступна.
* В системе существуют VIP-столики.

### Триггер

Клиент открывает страницу бронирования.

### Основной сценарий

1. Клиент открывает страницу бронирования.
2. Система загружает список VIP-столиков.
3. Система отображает доступные VIP-столики.
4. Клиент выбирает интересующий столик.
5. Система отображает доступные временные интервалы.
6. Клиент просматривает доступное время.

### Альтернативный сценарий

5а. Для выбранного столика отсутствуют свободные интервалы.
5а.1. Система отображает сообщение об отсутствии доступного времени.

### Постусловия

Клиент получил информацию о доступных VIP-столиках и свободном времени.


## UC-2. Создание бронирования

### Связанные функциональные требования

* FR-3. Создание бронирования
* FR-4. Проверка доступности
* FR-5. Предотвращение двойного бронирования
* FR-10. Уведомление клиента

### Связанные нефункциональные требования

* NFR-1. Время отклика системы не должно превышать 3 секунд
* NFR-2. Система должна поддерживать одновременную работу не менее 50 пользователей
* NFR-3. Система должна предотвращать создание двух бронирований одного столика на пересекающееся время
* NFR-5. Персональные данные клиентов должны храниться в защищённом виде
* NFR-6. Пользователь должен иметь возможность создать бронирование не более чем за 5 действий

### Актор

Клиент

### Цель

Создать бронирование VIP-столика.

### Предусловия

* Клиент выбрал VIP-столик.
* Для выбранного столика существует свободный временной интервал.

### Триггер

Клиент нажимает кнопку «Забронировать».

### Основной сценарий

1. Клиент выбирает VIP-столик.
2. Клиент выбирает дату бронирования.
3. Клиент выбирает время начала и окончания бронирования.
4. Клиент вводит контактные данные.
5. Система проверяет корректность введённых данных.
6. Система проверяет доступность столика.
7. Система создаёт запись о бронировании.
8. Система присваивает бронированию статус «Создано».
9. Система сохраняет бронирование.
10. Система отображает подтверждение успешного создания бронирования.
11. Система уведомляет клиента о создании бронирования.

### Альтернативные сценарии

#### A1. Столик уже занят

6а. Выбранный временной интервал недоступен.
6а.1. Система отображает сообщение об ошибке.
6а.2. Клиент выбирает другое время.

#### A2. Некорректные данные

5а. Система обнаруживает ошибки в заполнении формы.
5а.1. Система отображает список ошибок.
5а.2. Клиент исправляет данные.

### Постусловия

В системе создано новое бронирование со статусом «Создано».


## UC-3. Отмена бронирования

### Связанные функциональные требования

* FR-6. Отмена бронирования
* FR-10. Уведомление клиента

### Связанные нефункциональные требования

* NFR-1. Время отклика системы не должно превышать 3 секунд
* NFR-5. Персональные данные клиентов должны храниться в защищённом виде

### Актор

Клиент

### Цель

Отменить ранее созданное бронирование.

### Предусловия

* Бронирование существует.
* Клиент имеет доступ к своему бронированию.

### Триггер

Клиент нажимает кнопку «Отменить бронирование».

### Основной сценарий

1. Клиент открывает информацию о бронировании.
2. Клиент выбирает действие «Отменить бронирование».
3. Система отображает запрос на подтверждение действия.
4. Клиент подтверждает отмену.
5. Система изменяет статус бронирования на «Отменено».
6. Система сохраняет изменения.
7. Система уведомляет клиента об отмене бронирования.
8. Система отображает сообщение об успешной отмене.

### Альтернативный сценарий

4а. Клиент отказывается от отмены.
4а.1. Система закрывает окно подтверждения.
4а.2. Статус бронирования остаётся без изменений.

### Постусловия

Статус бронирования изменён на «Отменено».


## UC-4. Подтверждение бронирования

### Связанные функциональные требования

* FR-7. Подтверждение бронирования
* FR-9. Просмотр списка бронирований

### Связанные нефункциональные требования

* NFR-1. Время отклика системы не должно превышать 3 секунд
* NFR-5. Доступ к функциям подтверждения должен быть ограничен ролью менеджера

### Актор

Менеджер ресторана

### Цель

Подтвердить или отклонить бронирование клиента.

### Предусловия

* В системе существует бронирование со статусом «Создано».
* Менеджер авторизован в системе.

### Триггер

Менеджер открывает список новых бронирований.

### Основной сценарий

1. Менеджер открывает список бронирований.
2. Система отображает бронирования со статусом «Создано».
3. Менеджер выбирает бронирование.
4. Менеджер просматривает информацию о бронировании.
5. Менеджер подтверждает бронирование.
6. Система изменяет статус бронирования на «Подтверждено».
7. Система сохраняет изменения.
8. Система отображает результат операции.

### Альтернативный сценарий

5а. Менеджер отклоняет бронирование.
5а.1. Система изменяет статус бронирования на «Отменено».
5а.2. Система сохраняет изменения.

### Постусловия

Статус бронирования изменён на «Подтверждено» или «Отменено».
